## 2. فلتر الكلمات (Content Filter)

### 2.1 الملف: `src/lib/moderation.ts`

#### نقاط القوة:
- ✅ تطبيع النص العربي (توحيد الألف والياء والتاء المربوطة)
- ✅ قائمة تشمل بعض الكلمات العربية والإنجليزية المسيئة
- ✅ الفحص يتم server-side قبل حفظ المنتج

#### نقاط الضعف الحرجة:

##### أ. سهولة التجاوز (Bypass):
| الأسلوب | مثال | النتيجة |
|---------|------|---------|
| إضافة مسافات بين الحروف | `ك س` | 🟢 يتجاوز |
| رموز خاصة | `k.s`, `f*ck` | 🟢 يتجاوز |
| Leet speak | `s3x`, `sh1t`, `p0rn` | 🟢 يتجاوز |
| أحرف يونيكود متشابهة | `ѕех` (cyrillic) | 🟢 يتجاوز |
| تشكيل داخل الكلمة | `كَس` | 🟢 يتجاوز |
| كتابة عربية بحروف لاتينية (Franco) | `kos`, `zeb` | 🟢 يتجاوز |
| كلمات مركبة | إدخال كلمة محظورة داخل كلمة عادية | ⚠️ إيجابية خاطئة محتملة |

**السبب:** دالة `text.includes(norm)` تقارن حرفياً دون أي معالجة للرموز أو المسافات أو التشكيل.

##### ب. قائمة الكلمات غير مكتملة:

❌ **لا تغطي الفرنسية إطلاقاً** (مع أن المنصة تدعم الفرنسية):  
`pute`, `merde`, `salope`, `connard`, `enculé`, `bordel`, `cul`, `bite`, `chier`, `niquer`

❌ **لا تغطي الحسانية (اللهجة الموريتانية المحلية):**  
غياب كامل لأي كلمات باللهجة المحلية رغم أن السوق موريتاني

❌ **عدد الكلمات ضئيل جداً:** 30 كلمة فقط — أي منصة تجارية تحتاج 500+

❌ **لا تغطي الكلمات المراوغة (المحرفة):** `منيوكة`, `شرموطة`, `قحب`, `خنى`, `فحولة`

##### ج. إيجابيات خاطئة (False Positives):
كلمة `كس` ستظهر في كلمات عادية مثل: **كس**ب، **كس**الى، م**كس**ور، **كس**وة  
كلمة `فحل` كلمة عادية في بعض السياقات (فحل النخيل، فحل الخيل)

##### د. لا فلترة على:
- ❌ الرسائل بين المستخدمين (`messages`)
- ❌ التقييمات والمراجعات (`reviews`)
- ❌ أسماء المتاجر (`stores.name`)
- ❌ أسماء المستخدمين (`profiles.full_name`)
- ❌ تحديث المنتجات (الفلتر يطبق فقط عند الإنشاء)

## 3. فحص الصور (Image Safety)

### 3.1 الملف: `src/lib/moderation.ts` + `src/components/upload/ImageUploader.tsx`

#### ما يفعله النظام حالياً:
```typescript
// moderation.ts - validateImageFile()
1. يتحقق من امتداد الملف (.jpg, .jpeg, .png, .webp, .avif)
2. يتحقق من حجم الملف (أقصى 10 ميجابايت)
3. يتحقق من اسم الملف (يبحث عن كلمات مثل porn, nude, sex)
```

#### ⚠️ هذه ليست حماية حقيقية للصور! الأسباب:

1. **لا يوجد فحص لمحتوى الصورة نفسه**: يمكن رفع أي صورة إباحية/عنيفة باسم `product1.jpg` وستمر
2. **فحص اسم الملف ساذج**: `my_image.jpg` يتجاوز الفلتر بسهولة
3. **ImageUploader يستخدم accept="image/jpeg,image/png,image/webp"** — ولكن `validateImageFile` يقبل `.avif` كذلك (تناقض)
4. **Storage migration يسمح بـ `image/gif`** ولكن لا `validateImageFile` ولا `ImageUploader` يسمحان به (تناقض آخر)
5. **Storage migration يحد الحجم بـ 5MB** ولكن `validateImageFile` يسمح بـ 10MB (تناقض ثالث ⚠️)

#### اقتراحات:
- دمج Google Cloud Vision API / AWS Rekognition / NSFW.js لفحص محتوى الصور
- كحد أدنى: استخدام مكتبة مثل `sharp` لاستخراج metadata والتأكد من أن الملف صورة فعلاً
- توحيد القيود بين جميع الطبقات

## 4. الحماية ضد السبام (Spam Prevention)

### ❌ **غير موجودة إطلاقاً**

| الميزة | الحالة |
|--------|--------|
| Rate limiting على إنشاء المنتجات | ❌ غائب |
| CAPTCHA في نموذج البيع | ❌ غائب |
| حد أقصى للمنتجات لكل مستخدم/يوم | ❌ غائب |
| كشف المنتجات المكررة | ❌ غائب |
| فترة انتظار بين الإرسالات (cooldown) | ❌ غائب |
| حظر عناوين IP المسيئة | ❌ غائب |
| middleware rate limiting | ❌ غائب |

**النتيجة:** يمكن لمستخدم واحد إنشاء مئات المنتجات في دقائق وإغراق قاعدة البيانات.

## 5. مسار المراجعة (Moderation Workflow)

### 5.1 آلية العمل الحالية:

```
مستخدم ← إنشاء منتج ← status='pending' ← انتظار المراجعة ← مشرف يوافق/يرفض
```

#### نقاط القوة:
- ✅ المنتجات الجديدة تنتقل إلى حالة `pending` افتراضياً
- ✅ المنتجات غير النشطة لا تظهر في الواجهة العامة (RLS: `status = 'active'`)
- ✅ لوحة تحكم للمشرفين مع صفحة مخصصة للمراجعة
- ✅ صلاحيات المشرف محمية بـ RLS + فحص الدور في layout

#### نقاط الضعف:

1. **لا يوجد إشعار للبائع** عند الموافقة أو الرفض
2. **لا يوجد سبب للرفض** — المشرف يضغط "رفض" فقط دون تعليق
3. **لا يوجد `moderated_by` أو `moderated_at`** في جدول المنتجات — لا أثر تدقيقي
4. **لا يوجد تصنيف أو أولوية** للمنتجات المعلقة (الأقدم أولاً فقط)
5. **لا يمكن للمشرف طلب تعديلات** — إما موافقة أو رفض نهائي
6. **لا يوجد فلترة آلية للـ pending queue** — كل المنتجات تنتظر المراجعة البشرية، حتى السليمة
7. **غياب نظام الإبلاغ** — لا يمكن للمستخدمين الإبلاغ عن منتجات مسيئة
8. **لا يوجد `reported` status** في جدول المنتجات
9. **المشرف لا يرى سبب رفض الفلتر الآلي** عند المراجعة اليدوية

## 6. ثغرات أمنية إضافية

### 6.1 عدم فحص المحتوى عند التعديل
الفلتر `moderateText` يستدعى فقط في `createProductAction`. لا يوجد فحص في:
- تعديل المنتج (لا يوجد edit action أصلاً)
- تعديل المتجر
- تعديل الملف الشخصي
- إرسال الرسائل

### 6.2 عدم اتساق القيود
- `moderation.validateImageFile`: حجم 10MB
- `storage migration`: حجم 5MB (القيمة الفعلية المطبقة)
- `moderation.validateImageFile`: يقبل `.avif`
- `ImageUploader accept`: لا يقبل `.avif`

### 6.3 RLS — مشكلة في admin:
```sql
create policy "Sellers update their products"
  on public.products for update
  using (auth.uid() = seller_id)
```
هذه السياسة تمنع المشرف من تعديل منتجات الآخرين عبر API المباشر، لكن `approveProduct` و `rejectProduct` تستخدمان `supabase.from('products').update()` من server-side (مع service_role ضمنياً). هذا مقبول لكن لا يوجد تدقيق — أي خطأ في `cast<any>()` قد يفتح ثغرة.

### 6.4 لا middleware للحماية من هجمات عامة:
- لا CSRF protection مخصص
- لا rate limiting على مستوى الـ middleware
- لا security headers (CSP, HSTS, X-Frame-Options)

## 7. توصيات وتقوية (مصنفة حسب الأولوية)

### 🔴 أولوية عاجلة (خلال أسبوع):

1. **إعادة بناء فلتر الكلمات بالكامل:**
   - استخدام مكتبة احترافية مثل `bad-words` + قوائم مخصصة
   - إضافة French words (100+ كلمة)
   - إضافة Hassaniya slang (ضروري للسوق الموريتاني)
   - كشف التجاوز: إزالة المسافات والرموز قبل الفحص
   - استخدام regex مع حدود كلمات (`\b`) لمنع الإيجابيات الخاطئة
   - كشف Leet speak عن طريق normalizing إضافي

2. **إضافة rate limiting:**
   ```typescript
   // مثال باستخدام upstash/ratelimit أو類似
   - 3 منتجات كحد أقصى لكل مستخدم في الساعة
   - 10 منتجات كحد أقصى في اليوم
   - cooldown دقيقتين بين كل منتجين
   ```

3. **إضافة CAPTCHA:**
   - Turnstile (مجاني) أو hCaptcha في نموذج البيع

### 🟡 أولوية متوسطة (خلال أسبوعين):

4. **فحص محتوى الصور:**
   - استخدام NSFW.js (مكتبة مجانية، client-side)
   - أو Google Cloud Vision API للتأكد من عدم وجود محتوى إباحي
   - استخدام `sharp` لفحص أن الملف صورة حقيقية واستخراج metadata

5. **تحسين مسار المراجعة:**
   - إضافة `moderated_by`, `moderated_at`, `rejection_reason` إلى جدول المنتجات
   - إضافة حالة `changes_requested` للمنتجات التي تحتاج تعديل
   - إرسال إشعار للبائع بحالة المراجعة

6. **كشف التكرار:**
   - فحص العنوان والوصف ضد المنتجات الموجودة
   - منع إنشاء منتج مطابق لنفس المستخدم

### 🟢 أولوية منخفضة (خلال شهر):

7. **نظام إبلاغ متكامل:**
   - جدول `reports` (reporter_id, product_id, reason, status, created_at)
   - إضافة `reported` status للمنتجات المُبلغ عنها
   - إشعار المشرفين بالبلاغات الجديدة

8. **توسيع الفلترة:**
   - فحص المحتوى في الرسائل (قبل الإرسال)
   - فحص التقييمات والمراجعات
   - فحص أسماء المتاجر
   - فحص عند تعديل المنتج

9. **تحسين المرونة:**
   - جعل قائمة الكلمات قابلة للتحديث من قاعدة البيانات
   - إضافة تصنيفات (إباحي، عنيف، احتيالي، سبام)
   - تسجيل محاولات التجاوز للتحليل

10. **سياسات أمان عامة:**
    - إضافة Security Headers
    - CSRF tokens للأفعال الحساسة
    - CSP header لمنع XSS
